# Oblixa V10 Specification

This file defines the V10 product specification for Oblixa.

V10 completion is measured by shipped product behavior, durable data contracts, visible workflow outcomes, automation, tests, and telemetry. Documentation can explain shipped behavior, but documentation is not a completion dependency for any V10 requirement.

## 1. Release Contract

V10 is scoped to the contract operations behaviors that users must complete daily: intake, review, ownership, approvals, renewals, obligations, exceptions, evidence, reporting, search, settings, and job recovery.

V10 must deliver:

- one reliable path from contract intake to approved operational record
- one daily queue for user-owned work across contracts, approvals, renewals, obligations, exceptions, evidence, review, and failed jobs
- one contract record that shows status, next action, health, obligations, dates, evidence, exceptions, approvals, and audit history without route hunting
- one reporting layer that generates operational summaries from product state rather than manually assembled files
- one governance model for workspace mode, roles, feature eligibility, plan eligibility, and hidden modules
- one reliability model for imports, extraction, exports, reminders, reports, notifications, and automation

V10 must use existing navigation families unless a requirement cannot be implemented inside the current Home, Contracts, Review, Work, Renewals, Exceptions, Evidence, Reports, Settings, Advanced, or Assurance surfaces.

## 2. Product Objectives

V10 GA launch is eligible only when the objectives below have shipped as product behavior and pass the release-candidate measurements in section 2.2. Post-GA operational SLOs are measured after launch and do not determine GA eligibility.

- First contract activation: at least 80% of new workspaces with a valid upload or import must reach first generated work item within 10 minutes of file acceptance.
- Daily action clearance: at least 95% of user-owned actionable items must be reachable from Home, Work, command palette, or contract detail in two clicks or fewer from authenticated Home.
- Contract record trust: every contract detail page in the V10 release-candidate fixture set must identify next action, owner, critical dates, missing critical data, open exceptions, outstanding evidence, renewal posture, health score, and latest audit event above the first fold at 1440px by 900px.
- Operational completeness: every task, obligation, approval, renewal checkpoint, exception, evidence request, field-review item, failed job, and automation approval in the V10 release-candidate fixture set must expose owner state, status, due state, source object, and primary action.
- No orphan work: every unassigned actionable record in the V10 release-candidate fixture set must appear in the unassigned Work lens within 60 seconds of creation.
- Renewal prevention: 95% of contracts with approved renewal or notice dates inside 365 days must produce dated renewal posture and reminder eligibility within 5 minutes; every contract in the V10 release-candidate fixture set with missing required dates must show blocked reminder state and direct remediation.
- Evidence accountability: evidence requests must produce product-controlled follow-up on schedule: reminder at due date minus 3 days, reminder at due date, overdue state within 5 minutes after due date, owner notification within 1 hour after due date, and escalation work item within 24 hours after due date.
- Report reliability: 95% of standard report runs must complete or fail with retryable diagnostics within 2 minutes; 99% must complete or fail with retryable diagnostics within 10 minutes.
- Export reliability: 95% of exports up to 50,000 rows must complete or fail with retryable diagnostics within 2 minutes; every truncated export in the V10 release-candidate fixture set must identify selected row count, exported row count, and truncation reason.
- Search as router: command palette must return at least one eligible destination or recovery action for 95% of queries that exactly match an existing contract title, counterparty, account, saved view, work item title, report family, or settings label.
- In-app recoverability: every empty, loading, partial, failed, unauthorized, and plan-gated state in the V10 release-candidate fixture set must show the state reason and at least one valid next action or "no action available" explanation.
- Audit-ready operation: every visibility-affecting settings change, role change, owner change, field approval, evidence state change, automation action, report delivery, export, and destructive action in the V10 release-candidate fixture set must write a queryable audit event within 30 seconds.
- Product self-explanation: at least 18 of 20 pre-GA first-time usability participants must complete upload, field review, owner assignment, and first generated work item without opening help documentation or leaving the product; the same path must also pass 100 scripted first-time activation sessions.

### 2.1 User Outcome Targets

V10 must improve these user outcomes:

- faster activation: median time from accepted upload to first generated work item must be under 5 minutes, and p90 must be under 10 minutes
- fewer missed renewals: 99% of contracts with approved notice or renewal dates inside 365 days must generate reminder eligibility, reminder job attempt, or visible failed_retryable reminder diagnostic within 5 minutes of scheduled reminder time
- lower orphan work: unassigned actionable work must stay below 5% of open actionable work per active workspace
- fewer failed exports: export runs must have a failure rate below 2% excluding user-canceled jobs
- reliable report delivery: scheduled report runs must have a failure rate below 3% over a rolling 30-day window
- auditable evidence collection: every evidence request state transition in the V10 release-candidate fixture set must be visible in the source contract audit trail
- faster review throughput: median save-and-next field-review transition must complete in under 1 second after server acknowledgement
- lower overdue work: active workspaces with a seven-day preactivation baseline must reduce overdue actionable work by at least 25% within 30 days after V10 activation; workspaces without that baseline must keep overdue actionable work below 10% of open actionable work by day 30

### 2.2 Measurement Fixtures and Windows

V10 measurements use fixed fixtures and fixed windows.

`valid upload` means a PDF, DOCX, or TXT file under 25 MB with non-empty text content, no malware scan failure, and an authenticated workspace member as uploader.

`valid import` means a CSV import with required columns `title` and `counterparty`, at least one row, fewer than 10,000 rows, UTF-8 encoding, and no row-level parse error in more than 5% of rows.

`file acceptance` is the timestamp when upload or import validation succeeds and a durable import or extraction job id is created.

`new workspace` means a workspace created within the last 14 days with fewer than 5 accepted uploads or imports before the measured activation event.

`first-time activation session` means the first authenticated browser session for a user in a new workspace that starts at Home, upload, import, onboarding, or contract creation and has no previous approved required field by that user.

`V10 release-candidate fixture set` must include:

- 5 Core workspaces: empty, small with 10 contracts, medium with 1,000 contracts, large with 10,000 contracts, and degraded jobs with failed_retryable and partial jobs
- 3 Advanced workspaces: decisions-heavy, campaigns-heavy, and relationships-heavy
- 3 Assurance workspaces: findings-heavy, controls-heavy, and automation-approval-heavy
- at least 50 contracts across lifecycle statuses
- at least 10 contracts with missing required fields
- at least 10 contracts with approved renewal or notice dates inside 365 days
- at least 10 unassigned actionable items
- at least 10 overdue items
- at least 10 blocked items
- at least 10 evidence requests covering open, submitted, accepted, rejected, overdue, expired, and revoked-link states
- at least 10 report runs covering queued, running, succeeded, partial, failed_retryable, failed_terminal, and canceled states
- at least 10 export jobs covering succeeded, truncated, partial, failed_retryable, failed_terminal, and canceled states

The fixture set may contain more records than the minimum coverage list, but GA launch metric denominators must use the fixed sample sizes below.

`seeded medium workspace data` means the Core medium fixture: 1,000 contracts, 100 work items, 50 obligations, 25 approvals, 25 exceptions, 25 evidence requests, 25 report runs, 25 export jobs, 25 import or extraction jobs, 50 audit events, and 10 saved views.

GA launch metric sample sizes:

- activation: exactly 100 valid activation runs, with 50 valid uploads and 50 valid imports
- command palette search: exactly 200 exact-match queries, with at least 40 contract, 30 counterparty, 20 account, 30 saved-view, 40 work-item, 20 report-family, and 20 settings-label queries
- report reliability: exactly 100 standard report runs across the required Core report families
- export reliability: exactly 100 export jobs, including 20 truncated exports and 20 exports at 40,000 to 50,000 selected rows
- renewal reminders: exactly 100 scheduled reminder evaluations for contracts with approved renewal or notice dates inside 365 days
- evidence follow-up: exactly 100 evidence requests with due dates distributed across due-minus-3-days, due-now, overdue, and expired states
- work reachability: exactly 200 actionable work items covering every V10 work item type
- contract record trust: exactly 50 contract detail pages covering every lifecycle status and health band
- recoverability: exactly 50 empty, loading, partial, failed, unauthorized, and plan-gated state fixtures
- product self-explanation: exactly 20 pre-GA first-time usability participants plus 100 scripted first-time activation sessions

GA launch metric windows:

- activation, report, export, search, performance, job visibility, reminder, and evidence follow-up launch samples are measured before GA on release-candidate environments
- launch sample denominators are fixed before a metric run starts and must not be changed after the first run begins
- pre-GA first-time usability participant metrics exclude sessions with browser close, network offline, user idle for 15 minutes, or upload canceled before file acceptance
- launch failure-rate denominators exclude user-canceled jobs and provider outages only when the job shows failed_retryable or failed_terminal diagnostics inside the product within the required SLO window

Post-GA operational SLO windows:

- activation, report, export, search, performance, and job visibility SLOs are reported over a rolling 7-day window after GA launch
- scheduled report and reminder SLOs are reported over a rolling 30-day window after GA launch
- post-GA SLO misses must create workspace health diagnostics, but they do not retroactively block GA launch

## 3. Global Constraints

- V10 must not introduce a new top-level product family.
- V10 must not require replacing the existing Next.js, Supabase, Stripe, Resend, OpenAI, Vercel, Vitest, Playwright, or Semgrep stack.
- V10 must not widen access to Advanced, Assurance, automation, or paid-plan behavior without workspace-mode, role, feature, and plan eligibility checks.
- V10 must not rely on hidden capability growth. Any new capability must have a visible user path, a server-side authorization rule, and a regression gate.
- V10 must not store secrets or private contract text in client-exposed telemetry, logs, URLs, or public environment variables.
- V10 must preserve org isolation for every read, write, job, export, report, notification, external link, and API route.
- V10 must prefer shared primitives for status badges, empty states, queue cards, inline actions, field provenance, job visibility, audit events, and permission hints.
- V10 must keep Core complete for intake, review, ownership, work, renewals, exceptions, evidence, reports, and settings without requiring Advanced or Assurance mode.
- V10 must keep Advanced and Assurance additive: those modes can add context and automation, but they must not break the Core action path.
- V10 must treat documentation as optional support material. Product behavior, automated tests, instrumentation, and operational dashboards are the authoritative release evidence.

V10 scope is split into priority tiers.

P0 requirements must ship for V10 beta:

- quantified objectives in section 2
- activation state machine
- unified Work lenses and actions
- contract header, next action, health score, and audit summary
- field review, provenance, and data quality
- renewal posture from approved dates
- evidence request and external submission flow
- approval and exception actions
- command palette coverage for Core records and actions
- reports, exports, and job visibility for Core workflows
- settings, roles, plan eligibility, module visibility, and workspace health
- security, privacy, accessibility, and performance gates
- data and API contracts in section 5

P1 requirements must ship before V10 GA:

- counterparty, account, and relationship workspace summaries
- Advanced decisions, campaigns, simulations, and programs integrated with Work
- Assurance findings, controls, scorecards, playbooks, review boards, and health graph integrated with Work
- command palette coverage for Advanced and Assurance records
- notification coverage for Advanced and Assurance work

P2 requirements are eligible stretch work and must not block V10:

- additional report families beyond the ten required report families
- additional automation playbooks beyond the approval-gated automation contract
- additional relationship timeline visualizations beyond the required summary and actions
- predictive scoring beyond the deterministic health score formula
- custom workspace-defined work item types

P1 or P2 work may ship in V10 only when it satisfies the same authorization, audit, telemetry, accessibility, and performance rules as P0.

- every P0 requirement has shipped behavior

- P1 and P2 surfaces can be hidden, disabled, or marked outside beta scope without blocking beta
- no shipped P1 or P2 behavior violates P0 security, privacy, audit, eligibility, accessibility, telemetry, or performance rules

V10 GA means:

- every P0 requirement and every P1 requirement has shipped behavior

- every GA launch metric in section 2 is measured against the V10 release-candidate fixture set, fixed sample sizes, and launch windows
- post-GA operational SLO dashboards exist for every post-GA SLO in section 2.2

- completion evidence is product behavior, tests, telemetry, audit records, and operational dashboards, not standalone documentation

### 3.2 Exact Product Semantics

V10 terms have the following meanings.

- `due today`: due timestamp is greater than or equal to local day start and less than local day start plus 1 day for the workspace timezone.
- `due soon`: due timestamp is greater than or equal to local day start plus 1 day and less than local day start plus 8 days for the workspace timezone.
- `overdue`: due timestamp is earlier than local day start for date-only records, or earlier than current time for timestamped records.
- `blocked`: the item cannot reach its normal completion action until a missing dependency is resolved. Blocking dependencies are missing owner, missing required approved field, missing approved critical date, pending approval, rejected evidence needing resubmission, failed source job, revoked external link, or explicit `blocked` status.
- `high risk`: any record with critical severity, health score below 60, renewal notice deadline inside 30 days, overdue approval older than 7 days, overdue evidence older than 7 days, contract value at or above 250,000 USD-equivalent, or an open exception with severity `high` or `critical`.
- `large result set`: more than 100 visible rows, more than 1,000 matching rows, more than 50,000 export rows, or any query expected to return more than 2 MB of JSON before compression.
- `partial`: a job or workflow state where at least one durable output was created and at least one requested output failed, was skipped, or remains unavailable. A partial state must identify completed count, failed count, skipped count, and retryable count.
- `stale owner`: owner is missing, inactive, removed from the workspace, has not signed in for 45 days, or no longer has a role that can act on the item.
- `compatible items`: selected work items that share the same mutation type, target owner field, completion status model, authorization requirement, workspace id, and audit event type.
- `first generated work item`: the first durable task, obligation, approval, renewal checkpoint, evidence request, exception, field-review item, or failed-job action created from an uploaded or imported contract.
- `retryable diagnostics`: a failure response with diagnostic id, failure category, user-visible failure summary, retry eligibility, retry action when retryable, and support-safe detail that excludes private contract text.
- `source object`: the contract, field, obligation, approval, exception, evidence request, report run, export job, import job, extraction job, automation run, finding, control, campaign, or decision that caused the visible work item.

### 3.3 Contract Health Formula

Contract health score is an integer from 0 to 100.

Start at 100 and subtract:

- 20 points when any required activation field is missing
- 15 points when any critical date required for renewal posture is missing or unapproved
- 15 points when any linked approval, obligation, evidence request, task, or renewal checkpoint is overdue
- 15 points when an open high or critical exception exists
- 10 points when outstanding evidence exists and is not overdue
- 10 points when renewal notice deadline is inside 30 days and posture is not `notice sent`, `renewed`, `terminated`, or `completed`
- 10 points when owner is missing or stale
- 10 points when a linked import, extraction, export, report, reminder, notification, or automation job is failed or partial with retryable items
- 5 points when recommended fields are missing

The score floor is 0. The score must show every applied deduction and link each deduction to its source object when a source object exists.

Health bands:

- `healthy`: 85 to 100
- `watch`: 70 to 84
- `at risk`: 60 to 69
- `critical`: 0 to 59

### 3.4 Eligibility Rules

Workspace modes:

- `core`: Home, Contracts, Review, Work, Renewals, Exceptions, Evidence, Reports, Settings, upload, import, export, field review, reminders, notifications, and Core report families
- `advanced`: every Core capability plus Decisions, Campaigns, Programs, Relationships, account and counterparty workspaces, simulations, analytics, and Advanced report families
- `assurance`: every Advanced capability plus Findings, Control policies, Scorecards, Playbooks, Review boards, Autopilot, Segments, Program evolution, Health graph, and Assurance report families

Role permissions:

- `viewer`: read eligible records and reports, use command palette navigation, download exports only when export permission is explicitly granted
- `legal_reviewer`: viewer rights plus field review, evidence review, exception update, renewal posture update, and legal approval actions assigned to the user
- `finance_reviewer`: viewer rights plus approval, evidence, value, renewal, and finance report actions assigned to the user
- `editor`: legal reviewer rights plus create and update contracts, tasks, obligations, evidence requests, exceptions, exports, and reports
- `ops_manager`: editor rights plus team assignment, bulk compatible work actions, report subscriptions, and workspace health triage
- `manager`: ops manager rights plus Advanced decision and campaign management when workspace mode allows it
- `admin`: manager rights plus workspace mode, module visibility, billing, member roles, external link policy, notification defaults, and destructive actions

Feature eligibility is allowed only when all conditions are true:

- the workspace mode includes the feature family
- the user's role includes the action
- the module is not hidden
- the billing plan includes the feature family
- the target record belongs to the user's organization
- the target record is not archived or deleted unless the action is restore, audit read, or export history read

Plan eligibility:

- `trial`: Core capabilities for 14 days, 100 contracts, 5 members, 1,000 report rows per export, no scheduled reports, no Advanced, no Assurance
- `core`: Core capabilities, 10,000 contracts, 50 members, 50,000 report rows per export, scheduled Core reports
- `advanced`: Core plus Advanced capabilities, 100,000 contracts, 250 members, 250,000 report rows per export, scheduled Core and Advanced reports
- `assurance`: Core plus Advanced plus Assurance capabilities, 250,000 contracts, 1,000 members, 500,000 report rows per export, scheduled Core, Advanced, and Assurance reports
- `enterprise`: configured contract, member, and export limits stored in workspace billing settings; Enterprise cannot bypass role, org isolation, privacy, audit, or hidden-module rules

### 3.5 Privacy Rules

V10 privacy constraints are:

- client telemetry must not include raw contract text, uploaded file content, extracted clause text, evidence free text, notes, comments, external responder message text, secrets, API keys, tokens, or private URLs
- client telemetry must not include unredacted email addresses, phone numbers, physical addresses, bank details, tax ids, government ids, signature blocks, or personal names entered by external responders
- allowed client telemetry identifiers are UUIDs, numeric counts, enum values from this file, booleans, durations, row counts, HTTP status classes, feature keys, route templates, and salted hashes generated server-side
- server logs may include diagnostic ids, org ids, user ids, route templates, enum states, counts, timings, and sanitized error categories
- URLs must not contain raw contract title, counterparty, clause text, evidence text, email address, token secret, or uploaded file name
- external links must use opaque tokens, must expire, must be revocable, and must expose only the scoped request payload
- exports must neutralize CSV formula cells that start with `=`, `+`, `-`, `@`, tab, carriage return, or line feed
- audit events must identify actor, action, target type, target id, organization id, timestamp, outcome, and safe metadata; audit metadata must not contain raw document text or secrets

## 7. Non-Goals

V10 does not require:

- a new top-level navigation area
- a new product family
- replacing existing architecture
- replacing existing workspace modes
- moving Advanced or Assurance functionality into Core without eligibility controls
- public exposure of paid or hidden features
- manual report assembly outside the product
- documentation deliverables as proof of completion
- training material as a prerequisite for completing core workflows
- silent automation that changes operational state
- telemetry containing raw contract text or uploaded document content
