## Security checklist for new API routes

- **Full rubric:** See [docs/SECURITY_PASS_CHECKLIST.md](docs/SECURITY_PASS_CHECKLIST.md) (A–U: threat model, authz, crons, webhooks, CSP, dependencies, IR, etc.). **Quarterly sweep worksheet:** [docs/SECURITY_SWEEP_QUARTERLY_WORKSHEET.md](docs/SECURITY_SWEEP_QUARTERLY_WORKSHEET.md) (execution checklist + Org-only Appendix Z).

- **Auth:** `src/proxy.ts` does not protect `/api/*`. Every `route.ts` must authenticate (session, cron secret, capability token, etc.) and return 401/403 when appropriate.
- **Service role:** `createAdminClient()` bypasses RLS. Every query/mutation must filter by `organization_id` (or equivalent tenancy) from server-derived context—never trust client-supplied org IDs for authorization.
- **Inbound automation:** Use `isInboundAutomationAuthorized` from `src/lib/security/inbound-automation-token.ts` for Slack/email/integration callback routes; optional per-route env vars are documented in `.env.example`.
- **Regression tests:** Add or extend route tests for auth failures and, for org-scoped handlers, consider assertions in `src/app/api/security-org-scope-queries.test.ts` (or route-local tests) so IDOR-style regressions are caught in CI. New `route.ts` files should have a colocated `route.test.ts` or an entry in `scripts/api-route-test-allowlist.txt` if covered by a bundled test file.
- **Outbound copy and Sentry:** New email, Slack, or notification payloads should not introduce un-scrubbed PII into Sentry; extend `src/lib/observability/sentry-scrub.ts` (and client/server Sentry configs) when adding new breadcrumb or error field shapes.

## Operational UI (dashboard surfaces)

- **Primitives:** `src/lib/ui/operational-surface.ts` (tones, shell classes) and `src/components/ui/operational-summary-card.tsx` (`OperationalSummaryCard`, `OperationalSurfaceLinkCard`, `OperationalQueueRow`, `OperationalSectionHeader`, `OperationalMetricChip`).
- **Sections:** Use `ui-eyebrow` + `ui-section-title` (major sections may use `text-xl` on the title). At most one short `ui-muted-tight` line of supporting copy. Avoid “Why:” explanatory prose in queue rows; state the operational fact directly.
- **Metrics:** Prefer `OperationalSummaryCard` (use `variant="compact"` in dense grids). Put secondary facts in `OperationalMetricChip` rows or `breakdown`, not dot-joined sentences.
- **Hubs / shortcuts:** Use `OperationalSurfaceLinkCard` with an explicit action label (“View…”, “Open…”).
- **Tone:** Derive `OperationalTone` from real counts, severity, SLA breach, or failed runs—not decorative badges.
- **Auth / marketing / external token pages:** Keep shared radii and `ui-input` / `ui-card` tokens; do not use operational status badges unless tied to real server state.
- **Hygiene check:** `npm run audit:ui-operational` or `npm run audit:ui-operational:strict` (same as `npm run audit:ui-operational -- --strict`). The script flags, in `src/app/**/page.tsx` and `loading.tsx`: legacy `text-2xl font-semibold text-zinc-900` KPI tiles, `text-2xl` paired with `tabular-nums` on the same line, `Why:` prose in dashboard routes, and `<dd>` with `text-zinc-950` in `page.tsx` (hero-style description lists). It also flags `bg-white` in `src/components/layout/header.tsx` and `legal-footer.tsx` (Wave 2: use `bg-surface`). Prefer `OperationalSummaryCard` / `contract-hero-metrics` instead.
- **Focus:** Whole-card links and operational footer links use `ui-operational-focusable` (see `globals.css`). `ui-link` includes a visible `focus-visible` ring.

## Wave 2 UI (tokens, tables, modals, gates)

- **Surfaces:** Prefer `bg-canvas` for full-layout backdrops and `bg-surface` (or `ui-card` / `ui-panel`) for elevated panels. Dashboard route `page.tsx` files should not use raw `bg-white` for page chrome; use tokens. **`(dashboard)/layout.tsx`** applies **`.ui-page-stack`** to the main content wrapper so sibling sections share default vertical rhythm (pages may still use an inner `ui-page-stack` when they expose multiple top-level blocks). **`.ui-table-shell`** and **`.ui-toolbar`** use `border-[var(--border-subtle)]` and `bg-surface` (horizontal scroll lives on the shell).
- **Empty lists:** Primary queues/tables should use **`EmptyState`** (or the same copy/action pattern) instead of only a muted table row.
- **Modals / overlays:** Escape closes; opening moves focus inside (first control); closing returns focus to the trigger (`delete-contract-button`, mobile nav drawer, command palette).
- **Touch targets:** Dense filter chips and floating controls should meet at least ~40px hit area (`min-h-9`, padding) where feasible.
- **Heavy client blocks:** Use `next/dynamic` with `ssr: false` and a short loading line for large client-only visualizations (e.g. execution graph, health graph concentration).
- **SEO:** `(auth)/layout.tsx` and `external/layout.tsx` export default `metadata` (noindex). Root marketing metadata stays on `src/app/layout.tsx`.
- **E2E:** With `E2E_TEST_EMAIL` / `E2E_TEST_PASSWORD`, `e2e/authenticated.spec.ts` runs Axe on a **matrix** from [`e2e/authenticated-a11y-paths.ts`](e2e/authenticated-a11y-paths.ts) after explicit `goto`, a **skip-link → `#main-content` focus** check, **§22.2** primary-nav and **§22.6** cmd-K smoke (Core workspace: no Decisions/Campaigns/Assurance/**Relationships** in nav or palette; **Watchlists** also omitted from cmd-K as a §10.4 utility), **programs** URL coherence (Advanced fixture shows programs; Core redirects to `/dashboard`), **docs/refinement.md §10** reachability/redirect smokes (Core-detected via `/decisions` probe), **§20.1** contracts search must not emit `/decisions` or `/campaigns` links, and a **390×844 viewport** pass (document width overflow tolerance). **Optional refinement seeds (Core fixture):** set `E2E_REFINEMENT_ACCOUNT_KEY`, `E2E_REFINEMENT_COUNTERPARTY_KEY` (defaults used otherwise), and/or `E2E_REFINEMENT_DECISION_ID`, `E2E_REFINEMENT_CAMPAIGN_ID`, `E2E_REFINEMENT_FINDING_ID`, `E2E_REFINEMENT_CONTROL_POLICY_ID` to assert redirects for real rows. **Optional multi-workspace E2E:** when set, [`e2e/refinement-optional-fixtures.spec.ts`](e2e/refinement-optional-fixtures.spec.ts) runs Advanced (`E2E_ADVANCED_EMAIL` / `E2E_ADVANCED_PASSWORD`), Assurance (`E2E_ASSURANCE_*`), and Core non-admin utility redirects (`E2E_CORE_NON_ADMIN_*`); see `.env.example`. **Fixture assumptions:** keep `AUTHENTICATED_A11Y_AND_VIEWPORT_PATHS` aligned with docs/refinement.md §10.1 (paths that 404/403 in CI are skipped in the Axe loop). See the file for the authoritative path list (includes `/contracts/new`, `/contracts/bulk`, `/settings/billing`, `/dashboard/persona`, `/settings/health`, etc.).
  - The fixture account is typically a workspace **admin** (utility routes like `/contracts/maintenance` stay reachable); non-admin Core users are redirected from §10.4 utility layouts per `assertCoreUtilitySurfaceOrRedirect` (see `src/lib/product-surface/route-guard.test.ts`).
- **§16.1 Keyboard / batch:** Open the command palette with **Meta+K** (macOS) or **Ctrl+K** (Windows/Linux). Review and bulk flows use row selection + primary actions (e.g. review queue, bulk import); keep focus rings visible (`ui-link`, `ui-operational-focusable`).
- **Visual regression:** Playwright screenshot baselines are **not** used in this repo (flaky across environments); rely on Axe + manual review. If that changes, document the baseline policy here.
- **§22.7 polish (manual):** Walk Core primary routes in `docs/refinement.md` §7.1 order and confirm spacing, empty states, and operational cards match the Wave 2 / operational UI rules above (no hero KPI tiles on Core dashboards).

## Refinement release rules (docs/refinement.md)

- **§1 Purpose / §3 objectives:** Refinement improves coherence and visible quality **without** removing implemented capability. When you change dashboard, nav, cmd-K, or workspace surface code, skim [docs/refinement.md](docs/refinement.md) §1–§3 and match PR intent to the seven `REFINEMENT_OBJECTIVES` strings in `src/lib/product-surface/refinement-trace.ts` (also asserted in `src/lib/product-surface/refinement-contract.test.ts`). GitHub PRs use `.github/pull_request_template.md` as a reminder.
- **§19 API / observability:** Every `route.ts` under `src/app/api/**` authenticates (see Security checklist above). UI surface follows `loadProductSurfaceContext` + guards; avoid leaking Advanced/Assurance through nav payloads, cmd-K, or badges (`filterNavBadgesForSurface`, `resolver.ts`). PII scrubbing for Sentry: `src/lib/observability/sentry-scrub.ts` (keep aligned with client + server Sentry configs).
- **§20 global / header search:** Not implemented today; see **implementation status** at the top of [docs/refinement.md](docs/refinement.md) §20. When it ships, add workspace-mode–gated results (no Advanced/Assurance leakage on Core), then enable the skipped hook **“refinement §20 global/header search — E2E hook”** in `e2e/authenticated.spec.ts` and add real assertions. Contracts table scope remains `/contracts?search=` (§20.1 smoke in the same file).
- **§4.1 / §23 PR rubric:** Do not delete schemas, API routes, migrations, or advanced code paths to “simplify” navigation or IA—hide or gate only. If a PR removes `supabase/migrations/**` rows or drops `src/app/api/**` handlers, it needs an explicit non-refinement rationale (security/incident) and must not be justified as product-surface cleanup alone.
- **§23 Non-goals:** Refinement PRs that remove capability to “clean up IA” should be rejected; use `workspace_mode`, `nav-visibility`, and layouts instead.
- **§24 Final rule:** Prefer coherence and quality over exposing the full platform in Core; when adding features, default to progressive disclosure. New primary nav items must go through `buildProductSurfaceContext` / `ROUTE_INVENTORY` / `refinement-contract.test.ts` updates as appropriate.
- **Appendix B (header / mobile):** `src/components/layout/header.tsx` shows a **Utilities** link when the dashboard passes `showUtilitiesLink` from `src/lib/product-surface/more-index-visibility.ts` (empty `/more` on Core should hide it). `src/components/layout/sidebar.tsx` uses **Open utilities** on small screens; cmd-K entries stay aligned with `nav-visibility` + `resolver.ts`.
